
              TECHNICAL SPECIFICATION
Development of a Mobile Application for a Coffee Shop "Athena"
1. Introduction
This technical specification describes the development of a mobile information system for the coffee shop "Athena". The system is intended to automate customer service processes including digital menu access, order placement, payment processing via card or QR code, and table reservation management.
The restaurant and café industry is actively undergoing digital transformation. Modern customers expect fast, convenient, and contactless service. Traditional service models, which rely on manual billing, paper menus, and phone-based reservations, lead to delays, human errors, and increased staff workload.
The development of the AIS "Athena" mobile application directly addresses these problems and provides a competitive advantage in a technology-driven market.

3. Purpose and Goals of System Development
3.1 Functional Purpose
The system is intended for automation of customer service processes in the coffee shop "Athena": digital menu access, order placement and tracking, payment processing (card / QR code), table reservation, and automated sales reporting.
3.2 Operational Purpose
The application will allow customers to:
view the digital menu with categories, descriptions, and prices;
add items to the cart and place orders;
make payments via card or QR code;
reserve tables for a selected date and time.

The system will allow employees and administrators to:
cashier — process and confirm customer payments;
barista — view assigned orders and mark them as completed;
administrator — manage menu items and view sales reports.

4. Characteristics of the Object of Automation
The object of automation is the customer service process in the coffee shop "Athena".
Current state (AS-IS): The café operates with manual order recording via basic POS, manual calculation of totals, manual end-of-shift reconciliation, and report preparation in Excel. This leads to ~8–10 calculation errors per week and 35–40 minutes spent on report generation per shift.
The system automates the following business processes:
customer ordering (digital menu → cart → order confirmation);
payment processing (card / QR code via secure external API);
order routing to barista upon payment confirmation;
menu management by administrator;
automatic report generation at end of shift.

5.3 Business Processes (BP)
BP-01: Customer Order Process (TO-BE)
1. Customer opens the application and browses the digital menu.
2. Customer selects items and adds them to the cart.
3. Customer confirms the order.
4. The system creates the order and sends it to the cashier.
5. Cashier processes the payment (card / QR).
6. The system confirms payment and routes the order to the barista.
7. Barista prepares the order and marks it as completed.
8. Customer receives the order.

BP-02: Menu Management Process (TO-BE)
1. Administrator logs into the system.
2. Administrator opens the menu management section.
3. Administrator adds, edits, or deletes menu items (price, description, category, images).
4. The system saves changes immediately to the database.
5. Updated menu is visible to customers in real time.

BP-03: Report Generation Process (TO-BE)
1. At the end of the shift, the administrator opens the reports section.
2. The system automatically generates the daily sales report.
3. The report includes: total revenue, number of transactions, average bill, breakdown by categories.
4. The administrator reviews and exports the report.

5.6 Business Rules
Each order must be automatically assigned to a barista upon payment confirmation.
Menu items must have a valid price greater than zero.
Each payment must be linked to an existing order.
Only the administrator can add, edit, or delete menu items.
Reservations must be linked to an existing table.
Duplicate customer email addresses are not permitted.

6.2 Data Volume
Up to 1,000 registered customers.
Up to 1,000 orders stored and processed.
Up to 500 menu items across all categories.
Order data stored for at least 3 years for analytics.
User activity logs stored for at least 1 year.

6.3 Data Integrity
Email addresses must be unique across all customer records.
Menu item prices must be greater than zero.
Each order must reference an existing customer (FK constraint).
Each payment must reference a valid order (FK constraint).
Each reservation must reference an existing table (FK constraint).

6.4 Data Security
Passwords stored using secure hashing (SHA-256 or higher).
Payment processing via HTTPS with encrypted communication protocols.
Access to administrative functions restricted to authorized users.
Personal user data protected from unauthorized access.

The property of a requirement ensuring it describes exactly one function and cannot be divided into smaller independent parts.

A coffee shop employee who prepares drinks and food items according to orders received through the system.
Baseline
An approved and fixed version of a requirement or specification used as a reference for measuring future changes.
Business Process (BP)
A structured sequence of activities performed by actors to achieve a specific business outcome (e.g., order processing, report generation).
Business Requirement (BR)
A high-level statement of the organization goals or needs that defines what problem the system must solve. Example: increase the average bill by 15%.
Business Rule
A constraint or policy governing system behavior. Example: only an administrator can modify the menu.

A coffee shop employee responsible for confirming and processing customer payments within the system.
Clarity
The degree to which a requirement is clearly and unambiguously stated, leaving no room for different interpretations.
Completeness
The extent to which a requirement contains all information necessary for implementation and verification.

The number of users actively using the system at the same time. Required minimum in this project: 150 concurrent users.
Consistency
The absence of contradictions or conflicts between different requirements in the specification.
Customer
A visitor of the coffee shop who uses the mobile application to browse the menu, place orders, make payments, and reserve tables.
Data Integrity
The accuracy and consistency of data stored in the system (e.g., prices must be greater than zero, email addresses must be unique).
Data Model
A structured representation of data entities, their attributes, and the relationships between them in the system.
Data Security
Protection of system data from unauthorized access, modification, or disclosure using hashing, HTTPS, and access control mechanisms.
Database
A structured collection of data stored and managed electronically. In this project: Firebase Realtime Database / Firestore.
Entity
A real-world object represented in the system data model (e.g., Customer, Order, Payment, Table, Reservation).
Exception Flow
A sequence of actions that occurs when an error or unexpected situation arises during the execution of a use case.
Firebase
A cloud-based backend platform by Google used for real-time database storage, authentication, and application hosting.
Foreign Key (FK)
A field in a database table that references the Primary Key of another table, establishing a relationship between entities.
Functional Requirement (FR)
A specific function or behavior the system must perform. Example: the system shall display the menu when the customer opens the application.
Interview
A structured method of gathering requirements by directly questioning stakeholders about their needs, goals, and constraints.
Kano Model
A requirements classification method that categorizes features into Must-be, Performance, and Attractive types based on user satisfaction impact.
KPI (Key Performance Indicator)
A measurable metric used to evaluate progress toward a business goal (e.g., average bill amount, error rate, report generation time).
Maintainability
The ability of the system to be updated, modified, and repaired with minimal effort and downtime.
MoSCoW Method
A prioritization technique that classifies requirements into four groups: Must have, Should have, Could have, and Won't have.
Non-Functional Requirement (NFR)
A requirement that defines system quality characteristics such as performance, reliability, security, and usability.
Order
A set of selected menu items confirmed by a customer, automatically recorded and routed to the cashier and barista by the system.
Payment
A financial transaction made by a customer to complete an order, processed via card or QR code through a secure external API (Stripe, Google Pay).
Performance
A quality attribute defining system speed and efficiency. In this project: menu loads within 3 seconds, orders processed within 4 seconds.
Postconditions
The state of the system after successful or unsuccessful completion of a use case.
Preconditions
Conditions that must be satisfied before a use case can begin its execution.
Primary Key (PK)
A unique identifier for each record in a database table, ensuring every entity instance can be uniquely identified.
Prioritization
The process of ranking requirements by importance to determine implementation order. Methods used: MoSCoW, Kano, RICE, WSJF.
QR Code
A machine-readable code linked to a specific café table, used for table identification and payment initiation.
Quality Attribute
A measurable characteristic of the system that determines its overall quality: performance, reliability, security, usability, scalability, maintainability.
Reliability
The ability of the system to operate continuously without failures. Target: at least 99.5% uptime per month.
Requirement
A statement describing a capability, condition, or constraint the system must satisfy to meet stakeholder needs.
Reservation
A booking of a specific table for a selected date and time, created by a customer through the mobile application.
RICE Method
A quantitative prioritization formula: Score = (Reach x Impact x Confidence) / Effort. Used to rank requirements objectively.
Scalability
The ability of the system to handle an increasing number of users and requests without performance degradation.
Security
The ability of the system to protect user and payment data from unauthorized access through encryption and secure communication protocols.
Stakeholder
An individual or group with an interest in or affected by the system: Owner, Administrator, Customer, Cashier, Barista.
Technical Specification (TS)
A formal document defining the goals, requirements, scope, and constraints for software development, agreed upon by both customer and developer.
TO-BE
The desired future state of a business process after planned improvements and automation are implemented.
Traceability
The ability to link a requirement to its source and to all related artifacts such as design decisions and test cases.
Trigger
An event or condition that initiates the execution of a use case. Example: customer opens the application.
Uptime
The percentage of time the system is available and operational. Required: at least 99.5% per month.
Usability
The degree to which the system is easy and intuitive to use. Target: complete an order in no more than 3 actions within 60 seconds.
Use Case
A description of interactions between an actor and the system aimed at achieving a specific goal.
User Requirement (UR)
A description of a task that a user needs to perform using the system (e.g., View Menu, Place Order, Process Payment).
User Story
A short description of a system function from the user perspective: "As [role], I want [goal], so that [reason]".
Verifiability
The property of a requirement that allows its implementation to be confirmed through testing, measurement, or inspection.
WSJF (Weighted Shortest Job First)
A prioritization method that scores requirements by (Business Value + Time Criticality + Risk Reduction) / Job Size.
